# Pack: Commitment-Readiness Note - шаблон для оценки готовности роли к делегированию

**Назначение:** формальный способ решить "готов ли сотрудник взять роль/проект
самостоятельно" - вместо интуитивной оценки. Первый применённый пример - роль
Data Quality Expert (22.07.2026, см. learning-journal).

## Структура шаблона

### 1. Claim (утверждение, которое проверяется)
Одно предложение: "Сотрудник в роли [X] готов вести [проект/задачу] самостоятельно,
без постоянного вмешательства [руководителя/директора]".

### 2. Holons in play (какие холоны участвуют, не путать между собой)
- **Продукт** - что меняется (система/проект)
- **Агент** - кто меняет (сотрудник в роли)
- **Директор/надсистема** - роль, от которой передаётся ответственность

### 3. Evidence gap (что ещё нужно проверить перед подтверждением claim)
Список конкретных, проверяемых вопросов:
- Может ли диагностировать проблему без подсказки?
- Может ли сформулировать план (не просто исполнить готовую инструкцию)?
- Есть ли опыт успешных некритичных кейсов без эскалации?
- Понимает ли контекст "зачем" (роли/интересы стейкхолдеров), не только техническую часть?

### 4. Gate/decision condition (условие снятия ограничения)
Конкретные, измеримые критерии, при выполнении которых claim подтверждается
и ответственность передаётся полностью.

### 5. Заблокированная работа (что НЕ делать, пока условие не выполнено)
Явный список ограничений - какие типы задач/проектов НЕ передавать до подтверждения.

### 6. Смежные принципы (тактика на переходный период)
Что применять пока readiness ещё не достигнута - например: делать критичный шаг
самому + разбирать постфактум; фиксировать разобранные кейсы как community templates.

## Применение

Использовать при передаче любой роли/ответственности в команде - вместо
"чувствую, что пока не готов" фиксировать явно через структуру выше. Хранить
заполненные Notes в трекере команды как отдельную секцию "readiness по ролям".

## Связь с другими заметками IWE
- Развитие сотрудников vs сроки клиентов (19.07) - исходная неудовлетворённость,
  из которой родился этот pack
- Декомпозиция сложности (15.07) - критерий "приходит с решением, не с вопросом
  как" встроен в evidence gap
- Кейс-менеджмент + community templates (FPF-2, 20.07) - механизм накопления
  evidence для будущих Commitment-Readiness Notes
