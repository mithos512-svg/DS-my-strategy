
## Commitment-Readiness Note: роль Data Quality Expert (2026-07-22)

**Формат по FPF (ailev/FPF):** claim + evidence gap + gate/decision condition +
заблокированная работа, до подтверждения readiness.

---

### Claim (утверждение, которое проверяется)
Сотрудник в роли Data Quality Expert готов вести проект по улучшению/поддержке
качества данных (Ataccama Data Trust) самостоятельно, без постоянного погружения
директора в детали.

### Holons in play (какие холоны участвуют, не путать между собой)
1. **Продукт** - система Ataccama Data Trust (то, что меняется)
2. **Агент** - сотрудник в роли Data Quality Expert (кто меняет)
3. **Директор/надсистема** - роль, от которой передаётся ответственность

### Evidence gap (что ещё нужно проверить/увидеть, прежде чем считать claim подтверждённым)
- Может ли сотрудник самостоятельно диагностировать проблему качества данных
  без подсказки, с какой стороны начать?
- Может ли сформулировать план решения (не просто исполнить готовую инструкцию)?
- Есть ли опыт успешного прохождения хотя бы 1-2 некритичных кейсов без эскалации?
- Понимает ли сотрудник контекст "зачем" (roles/interests заказчика/клиента),
  не только техническую часть?

### Gate/decision condition (условие, при котором можно снять ограничение)
Проект передаётся сотруднику ПОЛНОСТЬЮ (без параллельного контроля директора),
когда:
- Пройдены минимум 2 некритичных кейса без эскалации к директору
- Сотрудник самостоятельно, без подсказки, декомпозировал хотя бы одну сложную
  задачу (критерий из S1, 15.07 - "приходит с вариантами решения, не с вопросом
  как сделать")
- Явно проговорена и подтверждена роль/интересы заказчика для конкретного проекта

### Заблокированная работа (что НЕ делаем, пока условие не выполнено)
- Не передавать проекты с жёсткими внешними дедлайнами без параллельного контроля
- Не снимать эскалацию к директору для новых, ранее не встречавшихся типов проблем

### Смежные принципы (что применить, пока идёт наращивание readiness)
- Рабочая гипотеза 19.07: делать критичный шаг самому в острых случаях +
  обязательно разбирать постфактум с сотрудником
- Кейс-менеджмент с community templates (FPF-2, 20.07): фиксировать разобранные
  кейсы как шаблоны - ускоряет накопление evidence для будущих claims
